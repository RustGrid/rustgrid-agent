# rustgrid-agent v1.1.0

This release replaces working-directory-dependent worker configuration with a
guided, host-aware setup flow.

## Install or upgrade

On macOS with Homebrew:

```sh
brew update
brew upgrade rustgrid-agent
```

For a new installation:

```sh
brew install RustGrid/tap/rustgrid-agent
```

## Guided setup

Run the interactive setup from any directory:

```sh
rustgrid-agent setup
```

The agent detects logical CPU and physical memory, recommends a balanced job
count, and derives the detailed Docker Sandbox resource settings. To select the
basic concurrency setting directly:

```sh
rustgrid-agent setup --max-concurrency 4
```

Setup writes a stable user configuration under
`$XDG_CONFIG_HOME/rustgrid-agent/config.json`, falling back to
`~/.config/rustgrid-agent/config.json`. Rerunning setup refreshes detected
capacity without replacing worker identity or login metadata. An existing
`.rustgrid-agent.json` in the launch directory is imported on first setup and
left untouched for rollback.

Explicit `--config` paths remain supported for managed deployments. When no
stable user configuration exists, commands continue to recognize a legacy
working-directory configuration for compatibility.

## Platform support

- macOS uses native system capacity APIs.
- Linux uses the process CPU allocation and `/proc/meminfo`.
- Production serving continues to require Docker Sandbox isolation and all
  existing readiness checks.

## Deployment status

Artifact publication is not production deployment approval. Complete staging
certification against the exact `v1.1.0` image digest before deploying it.

See [known limitations](https://github.com/RustGrid/rustgrid-agent/blob/v1.1.0/docs/known-limitations.md)
and [production operations](https://github.com/RustGrid/rustgrid-agent/blob/v1.1.0/docs/operations.md)
for the complete operational boundary.

## Rollback

Stop the worker and reinstall `v1.0.1`, then start it with an explicit legacy
configuration path if needed. Setup does not delete the imported legacy file.
Do not replace or mutate existing tags or release assets; publish further fixes
under a new semantic version.
